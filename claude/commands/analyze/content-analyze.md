# 콘텐츠 시스템 분석

$ARGUMENTS 시스템을 분석한다.

추론 금지. 코드에서 찾지 못하면 "확인 필요"로 표시.
출처는 파일명:라인번호 형식으로 명시.
출력 형식은 사용자에게 먼저 물어본다 (채팅 출력 / .md 파일 저장).

---

## 인자 파싱

`$ARGUMENTS`에서 아래 옵션을 파싱한다.

| 옵션 | 설명 | 기본값 |
|---|---|---|
| `--docs <목록>` | 생성할 문서를 콤마로 지정: `역기획서`, `작업가이드`, `단계출시`. `전체`(또는 `all`)로 3개 다 지정 가능 | **생략 시 역기획서만** |

예시:
- `인게임 로직` → 역기획서만 생성 (기본값)
- `인게임 로직 --docs 전체` → 3개 전체 생성
- `스킬 --docs 역기획서,작업가이드` → 역기획서 + 작업가이드 생성

`--docs`를 파싱한 뒤 나머지 텍스트를 콘텐츠명으로 사용한다.

---

## 프로젝트 설정 자동 감지

분석 시작 전 아래 항목을 파악한다. 파악이 불가능한 항목은 사용자에게 질문한다.

**SCRIPTS_PATH 우선순위**: `Docs/porting/PORTING_VOCAB.md`가 존재하면 해당 파일의 `{SCRIPTS_PATH}` 행 값을 사용한다. 없으면 find로 감지한다.

**연관그래프 우선참조**: `DATA_FILE`·`DATA_DECODE_CMD`·해금 조건 등 콘텐츠 관련 정보는 `Docs/design/{콘텐츠명}/{콘텐츠명}_연관그래프.md`(content-scan 산출물)가 존재하면 그 안에서 가져온다. 그래프가 없거나 낡았으면 재탐색하지 않고 `/analyze:content-scan {콘텐츠명}`을 먼저 실행하도록 안내한다.

| 설정 항목 | 감지 방법 |
|---|---|
| `SCRIPTS_PATH` | PORTING_VOCAB.md `{SCRIPTS_PATH}` 행 → 없으면 `find . -maxdepth 4 -type d -iname "*script*"` 결과 중 가장 포괄적인 경로 |
| `UI_PATH` | SCRIPTS_PATH 하위 `UI` 또는 `ui` 디렉토리 |
| `DATA_FILE` | `Docs/design/{콘텐츠명}/{콘텐츠명}_연관그래프.md`가 존재하면 헤더의 `DATA_FILE` 값을 사용. 없으면 `/analyze:content-scan {콘텐츠명}`을 먼저 실행하도록 안내 |
| `DATA_DECODE_CMD` | 연관그래프 헤더의 `DATA_DECODE_CMD` 값 사용. 없으면 `/analyze:content-scan {콘텐츠명}` 안내 |
| `CONDITIONS_FILE` | (해금 조건 전용 파일 개념 폐기 — 분석 항목 1의 "콘텐츠 해금 조건"에서 연관그래프의 카테고리 B 항목을 직접 참조한다) |
| `CONST_FILE` | `find SCRIPTS_PATH -name "Const.cs" -o -name "Constants.cs" -o -name "GameConst.cs"` |
| `BALANCING_FILE` | `find SCRIPTS_PATH -name "Balanc*.cs" -o -name "Formula*.cs"` |
| `LAUNCH_SPEC_DOC` | `find Docs/ -name "*런칭*" -o -name "*launch*spec*" -o -name "*통합*"` 상위 결과 |
| `REFERENCE_REVIEW_DOC` | 기존 단계출시 검토 문서 — `find Docs/ -name "*단계출시*검토*"` 첫 번째 결과 |
| `ANALYST_AGENT` | CLAUDE.md에 명시된 게임 분석 에이전트명 — 없으면 `general-purpose` |

감지 결과를 사용자에게 한 번에 보여주고 확인받은 뒤 분석을 시작한다.

## 파이프라인

```
STEP 1 (ANALYST_AGENT)       →  STEP 2 (메인 spot-check)  →  STEP 3 (불일치 수정)
  분석 + 지정 문서 작성           핵심 수치 grep 검증            파일 직접 수정
```

### STEP 1 — 분석 + 문서 작성 (ANALYST_AGENT)

**그래프 존재·신선 확인** (재실행 멱등성 게이트):
```bash
ls Docs/design/{콘텐츠명}/{콘텐츠명}_연관그래프.md 2>/dev/null && echo "EXISTS" || echo "NONE"
```
- 없음 → `/analyze:content-scan {콘텐츠명}`을 먼저 실행하도록 안내하고 대기
- 있음 → content-scan STEP 0과 동일한 방식으로 신선도 판정(기준 커밋 대비 `*.cs` 변경 0건이면 신선). 신선하면 **재탐색 없이 그래프를 Read해서 그대로 소비**한다 — `--docs`를 나눠 여러 번 실행하거나 STEP 3 수정 루프로 재진입해도 이 단계에서 매번 재탐색하지 않는다. 낡았으면 content-scan 재실행을 안내한다.

`Agent` 도구로 `ANALYST_AGENT`(46행에서 확정된 값)를 **`effort: max`**로 호출한다 — 기획자 관점 서술·해석 판단이 몰리는 단계라 정확도를 우선한다 (content-scan의 기계적 탐색 단계는 기본 effort를 쓰는 것과 대비됨).

`--docs`에 명시된 문서만 작성한다. 생략 시 역기획서만 작성.
각 문서 섹션의 분석 항목을 읽고 코드·데이터를 탐색하면서 해당 문서에 바로 작성한다. 연관그래프에 이미 있는 근거는 재탐색하지 않고 그대로 인용한다.

> 예: 카드 수를 확인할 때 상수 정의 파일(CONST_FILE)뿐 아니라 실제 카드를 꺼내는 UI 파일까지 추적.
> 예: 보상량을 확인할 때 테이블의 index뿐 아니라 실제 지급 시 호출하는 Balancing 함수(BALANCING_FILE)까지 추적.

게임 데이터 디코딩 (DATA_DECODE_CMD가 있는 경우):
```bash
{DATA_DECODE_CMD}           # 테이블 목록
{DATA_DECODE_CMD} {Table}   # 특정 테이블 출력
```

문서 저장 전 디렉토리가 없으면 `mkdir -p`로 먼저 생성한다.

분석 완료 후 채팅에는 **검증용 핵심 수치 목록**만 반환한다:
- 숫자로 표현되는 수치 5~10개 (최대값, 비용, 조건 등)
- 각 수치의 출처 파일명:라인번호
- 형식: `[수치명] 값: X (파일명:라인번호)`

### 역기획서 (`Docs/design/{콘텐츠명}/{콘텐츠명}_역기획서.md`)

**독자: 게임 기획자**
- 코드 경로·라인번호·클래스명은 기획 내용과 분리해서 작성
- 플레이어 관점으로 서술: 내부 enum 값 → 인게임 명칭, 조건 코드 → "XX 이상 진행 시"
- 재화·아이템 연관을 표로 정리 (획득 경로 → 사용처 → 보상)
- 레드닷·해금 조건은 게임 상황으로 설명
- 수치는 **실제 사용 값** 기준 (정의 위치 값이 아님)
- 확인되지 않은 항목은 "확인 필요"로 표시

**상단 참조 출처 명시** (제목 바로 아래)
```
참조 테이블: `TableA`, `TableB`
참조 코드: `ScriptA.cs`, `ScriptB.cs`
```
분석에서 실제로 읽은 테이블·파일만 나열한다.

#### 분석 항목

**1. 종류 및 현황** `[공통]`
- 현재 구현된 전체 목록 (데이터 테이블 또는 enum에서 추출)
- 각 항목의 해금 조건 (unlock_stage / unlock_level 또는 별도 조건 컬럼)
- 코드상 예약·미구현 슬롯 여부 (enum 빈 값, 주석 처리된 항목 등)
- **콘텐츠 해금 조건**: 연관그래프의 카테고리 B(조건 함수/트리거) 항목을 확인한다 — 플래그형(특정 함수 호출)인지 진행도 스탯 임계값형(`레벨/티어 > N`)인지 그래프에 이미 분류돼 있다. 그래프에 B 항목이 없거나 `확인 필요`로 남아있으면 해당 위치를 직접 Read해 확인한다.

**2. 강화·성장 구조** `[조건부 — 1번에서 확인된 강화 축(레벨/그레이드/단계)이 있는 콘텐츠만 해당. 없으면 이 항목 생략]`
- 강화 축 (레벨/그레이드/단계 등) 및 최대치 → CONST_FILE에서 이 콘텐츠의 최대치 상수를 찾는다 (접두사를 하드코딩하지 않고, 콘텐츠명·강화 축 이름과 연관된 상수명을 CONST_FILE 안에서 탐색). 연관그래프에 카테고리 A(테이블 타입 연결) 항목으로 이미 참조돼 있으면 그 파일:라인을 그대로 쓴다.
- 강화 비용 산식 → BALANCING_FILE 관련 함수
- 강화로 변하는 수치 및 효과

**3. 연관 시스템** `[공통 — 연관그래프의 카테고리를 그대로 옮겨 적는다]`
연관그래프(`Docs/design/{콘텐츠명}/{콘텐츠명}_연관그래프.md`)에 있는 카테고리 노드를 아래 대응표로 옮겨 적는다. 그래프에 없는 카테고리는 "해당 없음"으로, `확인 필요`로 남은 항목은 그대로 기재한다 — 임의로 "없음" 단정 금지(매핑 규칙 문서의 확인 필요 처리 경계 참고).

| 그래프 카테고리 | 역기획서 서술 |
|---|---|
| A 테이블 타입 연결 | 데이터 테이블·enum 체계 등록 여부 |
| B 조건 함수/트리거 | 해금·노출 조건 (1번 항목과 중복되면 생략하고 1번만 표기) |
| C 테이블 존재 연동 | 퀘스트 등 다른 콘텐츠의 목표·보상 연동 여부 |
| D 로그 이벤트 | 분석 로그 수집 여부 |
| E 메인 루프 진입 | 핵심 게임 루프 포함 여부 (직접 호출·이벤트/delegate 연결 포함, switch-case로 한정하지 않음) |
| F 실결제 IAP | 스토어 상품 연동 여부, 전용 팝업 존재 여부 |
| G-1 자동 노출 팝업 | 특정 조건에서 자동으로 열리는 팝업 |
| G-2 메뉴 진입형 팝업 | 사용자가 버튼으로 여는 일반 패널 |
| H 버프/모디파이어 | 이 콘텐츠가 다른 시스템의 수치에 주는 영향 |
| I 저장 데이터 포함 | 진행 상태의 영구 저장 여부 |
| J 치트/디버그 연동 | QA·치트로 조작 가능 여부 |
| K 알림 뱃지 연동 | 레드닷 등 신규/보상대기 알림 표시 여부 |

**4. UI 구성** `[공통]`
- 관련 Popup / Fragment 파일 목록 (find로 탐색)
- 화면에 노출되는 UI 요소 목록과 각 요소의 역할 (버튼·텍스트·아이콘 등, 플레이어가 보는 화면 기준 — 코드 연결 방식은 업데이트 작업가이드 5-1 항목에서 다룸)
- 보유/미보유 상태 분기 여부
- **레드닷 연동**: 3번 항목의 카테고리 K(알림 뱃지 연동)를 참고

### 업데이트 작업가이드 (`Docs/programmer/{콘텐츠명}/{콘텐츠명}_업데이트_작업가이드.md`)

**독자: 프로그래머**
- 파일명:라인번호 + 작업 내용 명시
- 신규 항목 추가 시 수정 위치 전체 (코드·테이블·에셋)
- 단계 출시 방식 선택 시 추가 작업 별도 섹션
- 작업 순서 체크리스트 포함

**상단 참조 출처 명시** (제목 바로 아래) — 역기획서와 동일 형식

#### 분석 항목

**5. 업데이트 작업 범위** `[공통]`

신규 항목 추가 시 수정해야 할 위치 전체:
- 코드: 파일명:라인번호 + 작업 내용
- 데이터 테이블: 추가해야 할 행 및 컬럼
- 에셋: 스프라이트 아틀라스 키, Addressable Prefab 키, 로컬라이즈 키

**5-1. 단계 출시 방식 선택** `[조건부 — 순차 오픈이 필요한 콘텐츠만 해당. 전량 동시 오픈이면 생략]`

콘텐츠를 한 번에 전량 오픈하지 않고 순차 오픈할 경우, 아래 3가지 방식 중 이 콘텐츠에 적합한 방식을 분석해서 작업가이드에 포함한다.

| 방식 | 설명 | 빌드 필요 | 적합한 경우 |
|---|---|---|---|
| A. 전처리문(`#if`) | 빌드 시 define symbol로 오픈 종류 제어 | 매월 필요 | 오픈 시점에 코드 동작 자체가 달라야 할 때 |
| B. 데이터 컬럼(`enabled`) | 데이터 테이블에 `enabled : bool` 컬럼 추가, 오픈 시 `true`로 변경 | 최초 1회 | 항목 목록이 데이터 기반이고 코드 변경 없이 제어 가능한 경우 |
| C. 데이터 파일 교체 | 출시 버전별 데이터 파일을 별도 관리하여 교체 | 불필요 | 테이블 구조 자체가 바뀌는 경우 |

각 방식 분석 시 확인 항목:
- 콘텐츠 목록을 반환하는 함수(`GetXxxTypes()` 등)가 데이터 기반인지 enum 하드코딩인지
- 해금 트리거(연관그래프의 카테고리 B 항목이 가리키는 실제 순회·체크 로직)가 disabled 항목을 건너뛰는지
- UI 슬롯이 Inspector 직접 할당인지 데이터 기반인지 → 필터링 시 None 처리 필요 여부
- 튜토리얼 `First()` / `Last()` 참조 여부 → 필터링 시 타겟 변경 가능성
- 퀘스트 카운트 로직이 전체 항목 기준인지 보유 항목 기준인지

### 단계 출시 검토 (`Docs/design/{콘텐츠명}/{콘텐츠명}_단계출시_검토.md`)

**독자: 게임 기획자**
- LAUNCH_SPEC_DOC의 월차별 오픈 계획과 교차 검증
- 구현상 제약(하드코딩, 빌드 필요 여부 등)을 기획자 언어로 서술
- 작업 방법(단계 출시 방식 A/B/C 비교, 코드 수정 위치)은 작업가이드 소속 — 이 문서에 포함하지 않음

**상단 참조 출처 명시** (제목 바로 아래) — 역기획서와 동일 형식

#### 분석 항목

**6. 단계 출시 검토** `[조건부 — LAUNCH_SPEC_DOC이 존재하고 이 콘텐츠가 해당 계획에 포함된 경우만 해당]`

LAUNCH_SPEC_DOC의 월차별 오픈 계획과 교차 확인하여, 계획대로 진행했을 때 오류가 발생하거나 기획 결정이 필요한 항목을 정리한다.

REFERENCE_REVIEW_DOC가 있으면 해당 문서를 구조 레퍼런스로 사용한다.

1. **기획자 검토 필요 항목** 요약
   - ✅ 확정된 사항
   - ❓ 결정 필요
   - ⚠️ 개발팀 전달 필요
2. **월차별 오픈 계획 요약** 테이블 (시점 / 오픈 콘텐츠 / 빌드 필요 여부)
3. **[이슈 #N]** 형식으로 각 이슈 상세
   - **문제**: 어떤 상황에서 문제가 발생하는가
   - **원인**: 코드/데이터 근거 (파일명:라인번호 허용)
   - **수정 추천**: 해결 방법 제안
4. **작업 분류 요약** 테이블 (시점 / 작업 종류 / 내용)

### STEP 2 — 핵심 수치 spot-check (메인에서 직접 수행)

STEP 1이 반환한 핵심 수치 목록을 grep/Read로 1:1 대조한다.

판정: **일치 / 불일치 / 확인 필요**
불일치 시: 실제 값 + 수정 대상 파일·섹션 명시

### STEP 3 — 불일치 수정 (메인에서 직접 수행)

STEP 2에서 불일치 항목이 있으면 해당 문서 파일을 직접 Edit으로 수정한다.
완료 후 저장된 파일 경로를 사용자에게 알린다.
