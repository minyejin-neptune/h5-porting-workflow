# GitHub 이슈 생성

아래 내용을 잘 구조화된 GitHub 이슈로 만들어 등록한다.

> 이슈 내용: **$ARGUMENTS**

인자가 `--no-confirm`으로 시작하면 그 플래그를 떼어내고 나머지를 이슈 내용으로 쓴다 — Step 1의 repo 확인을 건너뛴다(서브에이전트 등 사용자에게 물어볼 수 없는 호출자용). 플래그 없으면 기존대로 확인한다.

인자(플래그 제외)가 비어 있으면 여기서 멈추고 "생성할 이슈 내용을 인자로 입력해주세요 (예: `/common:create-issue 로그인 화면에서 새로고침하면 세션이 풀림`)" 라고 안내한다.

---

## 원칙 (엄수)

- **추론 금지 표기**: 코드에서 확인한 사실은 파일:라인 근거를 붙인다. 확인 못 한 내용은 `[추정]` 또는 `[확인 필요]`로 표기하고, 사실인 것처럼 단정하지 않는다.
- **불충분하면 질문**: 제목·본문을 채우기에 정보가 부족하면 임의로 지어내지 말고 1~3개 확인 질문을 먼저 한다.

---

## Step 1 — 사전 확인

```bash
gh auth status            # 인증 여부
gh repo view --json nameWithOwner -q .nameWithOwner   # 대상 repo 확인
```

- 인증이 안 돼 있으면 멈추고 "`! gh auth login` 으로 로그인해주세요" 안내.
- repo가 감지되면 그 repo에 생성한다. `--no-confirm`이면 바로 진행, 아니면 현재 위치가 대상 repo가 맞는지 사용자에게 한 번 확인한다.
- 라벨을 붙일 계획이면 존재 여부를 확인한다: `gh label list`.

## Step 2 — 유형 판별

내용을 보고 이슈 유형을 판별해 제목 접두어와 본문 틀을 고른다.

- **Bug** — 무언가 오작동/에러. 현상·재현·기대동작이 핵심.
- **Feature** — 새 기능/개선 요청. 배경·제안·완료조건이 핵심.
- **Task/Chore** — 작업 항목(리팩터·문서·설정 등).

유형이 애매하면 사용자에게 물어본다.

## Step 3 — 이슈 초안 작성

### 제목
- 한 줄, 명확하게. 무엇이 어디서 문제인지/무엇을 원하는지 드러나게.
- 필요 시 접두어: `[Bug]`, `[Feature]`, `[Task]`.

### 본문 (유형별)

**Bug**
```
## 현상
<무슨 일이 일어나는가>

## 재현 절차
1. ...
2. ...

## 기대 동작
<원래 어떻게 되어야 하는가>

## 환경 / 근거
- <버전·플랫폼 등, 아는 것만>
- 관련 코드: <파일:라인 | [확인 필요]>
```

**Feature / Task**
```
## 배경 / 문제
<왜 필요한가>

## 제안
<무엇을 하려는가>

## 완료 조건 (DoD)
- [ ] ...
- [ ] ...

## 참고
- 관련 코드/문서: <파일:라인 | [추정]>
```

## Step 4 — 생성

Step 3에서 작성한 제목·본문으로 바로 생성한다. 본문은 임시 파일로 전달해 서식이 깨지지 않게 한다.

```bash
BODY_FILE="$(mktemp)"
cat > "$BODY_FILE" <<'EOF'
<작성한 본문>
EOF

gh issue create \
  --title "<작성한 제목>" \
  --body-file "$BODY_FILE"
# 선택: --label "<라벨>" --assignee "<@사용자>" --milestone "<마일스톤>"

rm -f "$BODY_FILE"
```

생성 후 반환된 **이슈 URL**을 사용자에게 보고한다. 실패하면(권한·라벨 없음 등) 원인을 그대로 전달하고 임의로 우회하지 않는다.

---

## 마무리

- 여러 이슈로 나누는 게 맞다고 판단되면 생성 전에 사용자에게 제안한다 (필요 시 `/common:feature-breakdown`으로 먼저 분해).
- 요청 범위 밖의 라벨 생성·프로젝트 연결 등은 임의로 하지 않는다.
