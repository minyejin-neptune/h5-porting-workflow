# 이슈 자동 해결 루프

GitHub 이슈를 가져와 **계획을 사람에게 검토받고**, 승인되면 **구현→검증→수정을 스스로 반복**해 해결한다.

> 대상 이슈 번호: **$ARGUMENTS**

인자가 비어 있거나 숫자가 아니면 멈추고 "해결할 이슈 번호를 입력해주세요 (예: `/common:auto-resolve 42`)" 라고 안내한다.

---

## 원칙 (엄수)

- **계획 검토 게이트는 필수**: 조사·계획까지 끝내면 **반드시 멈추고 사용자 승인을 받는다.** 승인 전에는 어떤 파일도 수정하지 않는다. (사람 개입 지점은 여기 하나)
- **검증 신호 없이는 루프 금지**: 통과/실패를 기계적으로 판정할 방법(빌드·컴파일·테스트·검증 스크립트)을 먼저 확정한다. 신호를 못 찾으면 **임의로 "됐다"고 하지 말고 사용자에게 검증 방법을 묻는다.** 자동 루프는 이 신호로만 종료된다.
- **추론 금지**: 원인·수정 근거는 코드에서 확인해 파일:라인으로 남긴다. 확인 못 하면 "코드에서 근거를 찾지 못했습니다"라고 명시하고 `[확인 필요]` 표기.
- **범위 고정**: 승인된 계획 범위 안에서만 수정한다. 루프 중 계획 밖 변경이 필요해지면 멈추고 사용자에게 보고한다.

---

## Phase A — 계획 (사람 검토까지)

### A1. 이슈 가져오기
```bash
gh auth status >/dev/null 2>&1 || echo "로그인 필요: ! gh auth login"
gh issue view $ARGUMENTS --json number,title,state,body,labels,comments,assignees,url
```
- 없거나 접근 불가면 멈추고 보고. **본문 + 코멘트**를 모두 읽어 요지를 2~3줄로 요약한다.

### A2. 코드베이스 조사
- 이슈가 지목한 파일·함수·에러를 grep/find로 추적한다(사용 명령·결과를 근거로 남김).
- 원인 위치와 손댈 지점을 **파일:라인**으로 특정한다. 못 찾으면 `[확인 필요]`.

### A3. 검증 방법 확정 (종료 신호 정의) — 중요
루프의 통과 판정에 쓸 명령을 이 프로젝트에서 찾는다. 하드코딩하지 말고 실제 존재를 확인한다.
```bash
ls package.json 2>/dev/null && grep -A3 '"scripts"' package.json   # npm test/build
ls *.sln *.csproj 2>/dev/null                                       # dotnet build/test
ls ~/github/h5-porting-workflow/templates/scripts/h5-port-verify.py 2>/dev/null  # H5 검증
```
- 후보(예: `dotnet build`, `npm test`, `h5-port-verify.py`, Unity 컴파일 로그 감시)를 **1개 이상 확정**한다.
- 못 찾으면 진행하지 말고 "이 이슈의 해결 여부를 무엇으로 판정하면 될까요?"라고 사용자에게 묻는다.

### A4. 계획 제시 + 검토 요청 (게이트)
아래 형식으로 제시하고 **승인을 기다린다. 승인 전 구현 금지.**
```
## 이슈 #$ARGUMENTS — <제목>
<요지> / 상태: <...> / URL: <...>

## 원인 분석
- <원인> — 근거: <파일:라인 | [확인 필요]>

## 해결 계획
1. <무엇을> — 파일: <경로:라인> — 이유: <...>
2. ...

## 검증 방법 (루프 종료 신호)
- 통과 판정 명령: <예: dotnet build && npm test>
- 재현/확인 시나리오: <...>

## 최대 반복 / 리스크
- 최대 반복: <기본 5회>
- 영향 범위·리스크: <있으면 / "특이사항 없음">
```
마지막에 묻는다: **"이 계획대로 자동 진행할까요? (승인하면 구현→검증→수정 루프를 돌립니다)"**

## Phase B — 자율 구현 루프 (승인 후에만)

작업용 브랜치를 먼저 만들어 되돌리기 가능하게 한다(git repo인 경우).
```bash
git rev-parse --is-inside-work-tree >/dev/null 2>&1 && git switch -c "fix/issue-$ARGUMENTS" 2>/dev/null || echo "git repo 아님 — 브랜치 생략, 변경 전 상태만 보고"
```

그리고 아래를 **종료 조건까지 반복**한다:

```
반복 n = 1..MAX(기본 5):
  1) Act    — 계획의 다음 단계를 구현 (Edit). 한 번에 작은 단위로.
  2) Evaluate— A3에서 정한 검증 명령 실행
  3) Observe — 출력(컴파일 에러·테스트 실패·로그)을 읽는다
  4) 판정:
       - 통과        → 루프 종료(성공)
       - 실패        → 로그에서 원인을 파일:라인으로 특정 → 다음 반복에서 수정
       - 진전 없음   → 같은 에러가 2회 연속이면 스스로 못 푸는 것 → 루프 중단, 사용자에게 상황 보고
  매 반복마다 "n회차: 무엇을 바꿈 / 검증 결과"를 한 줄 로그로 남긴다.
```

**종료 조건**: `검증 통과` OR `최대 반복 도달` OR `연속 2회 진전 없음`. 통과가 아닌 종료는 실패로 명확히 보고한다(성공으로 포장 금지).

## Phase C — 결과 보고

```bash
git rev-parse --is-inside-work-tree >/dev/null 2>&1 && git --no-pager diff --stat HEAD 2>/dev/null
```
아래를 사용자에게 보고한다:
- **결과**: 성공 / 실패(사유)
- **변경 파일 요약**: diff --stat
- **검증 결과**: 마지막 검증 명령의 실제 출력
- **반복 로그**: 몇 회 만에 통과했는지

커밋/머지는 사용자 확인 후 진행한다. (PR은 이 워크플로우에선 선택 — 필요하면 사용자가 지시)

---

## 마무리

- 계획 검토(Phase A4)와 최종 보고(Phase C)가 유일한 사람 개입 지점이다. 그 사이 구현·검증·수정은 자동으로 돈다.
- 큐 방식으로 여러 이슈를 자동 처리하려면 `/loop` 로 이 커맨드를 이슈 번호별로 반복 호출한다.
- 규모가 큰 이슈는 Phase A2 전에 `/common:feature-breakdown`으로 먼저 분해한다.
