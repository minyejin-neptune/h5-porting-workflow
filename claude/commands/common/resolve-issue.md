# 이슈 해결

GitHub 이슈를 가져와 **해결을 위한 상세 계획**을 세우고, 계획을 제시한 뒤 바로 구현까지 진행한다.

> 대상 이슈 번호: **$ARGUMENTS**

인자가 비어 있거나 숫자가 아니면 멈추고 "해결할 이슈 번호를 입력해주세요 (예: `/common:resolve-issue 42`)" 라고 안내한다.

---

## 원칙 (엄수)

- **추론 금지**: 계획의 각 단계는 코드에서 직접 확인한 근거(파일:라인)에 기반한다. 근거 없이 필요하다고 본 단계는 `[추정]`, 확인이 필요한 부분은 `[확인 필요]`로 표기한다. 코드에서 못 찾으면 "코드에서 근거를 찾지 못했습니다"라고 명시한다.
- **불충분하면 질문**: 이슈 내용이 모호해 계획을 못 세우면 임의 가정 없이 먼저 확인 질문을 한다.

---

## Step 1 — 이슈 가져오기

```bash
gh auth status >/dev/null 2>&1 || echo "로그인 필요: ! gh auth login"
gh issue view $ARGUMENTS --json number,title,state,body,labels,comments,assignees,url
```

- 이슈가 없거나 접근 불가면 멈추고 원인을 보고한다.
- **제목·본문·라벨·코멘트**를 모두 읽는다. 코멘트에 추가 맥락/결정이 있을 수 있으므로 반드시 확인한다.
- 이슈 요지를 2~3줄로 요약해 보고한다.

## Step 2 — 코드베이스 조사

이슈가 지목하는(또는 관련된) 코드를 실제로 찾아 읽는다.

- 이슈에 언급된 파일·함수·에러 메시지를 grep/find로 추적한다. 사용한 명령과 결과를 근거로 남긴다.
- 문제의 **원인 위치**와 **손대야 할 지점**을 파일:라인으로 특정한다.
- 원인을 코드에서 확인하지 못하면 단정하지 말고 `[확인 필요]`로 남기고 무엇을 더 봐야 하는지 적는다.

## Step 3 — 해결 계획 수립

원인 분석을 바탕으로 실행 계획을 만든다. 규모가 크면 하위 작업으로 나눈다(필요 시 `/common:feature-breakdown` 활용).

각 단계에 다음을 채운다:
- **무엇을** — 어떤 파일에 어떤 변경을
- **왜** — 이슈의 어느 부분을 해결하는가
- **근거** — 파일:라인
- **완료 조건** — 무엇을 보면 해결됐다고 판단하는가 (테스트/재현 시나리오 포함)

리스크·부작용(다른 코드 영향, 마이그레이션 등)과 대안이 있으면 함께 적는다.

## Step 4 — 계획 제시

아래 형식으로 계획을 제시한다. 승인을 기다리지 않고 Step 5로 바로 이어간다.

```
## 이슈 #$ARGUMENTS — <제목>
<요지 2~3줄> / 상태: <open/closed> / URL: <...>

## 원인 분석
- <원인> — 근거: <파일:라인 | [확인 필요]>

## 해결 계획
1. <무엇을> — 파일: <경로:라인> — 이유: <...>
2. ...

## 검증 방법
- <어떻게 해결을 확인할지: 재현 절차·테스트>

## 리스크 / 영향 범위
- <있으면 / 없으면 "특이사항 없음">

## 미해결 / 확인 필요
- <있으면>
```

## Step 5 — 구현

Step 4의 계획대로 코드를 수정한다. 계획의 각 단계를 실제로 실행하고, 완료 조건을 검증한다.

- 이슈에 DoD 체크박스가 있으면 완료된 항목을 `gh issue edit`로 즉시 갱신한다.
- 구현 중 계획과 다르게 처리해야 하는 경우가 생기면(범위 변경, 대안 채택 등) 이슈 본문을 먼저 갱신한 뒤 진행한다.
- 완료 후 커밋 메시지에 `(#$ARGUMENTS)`를 참조한다.

---

## 마무리

계획 제시 후 구현까지 한 번에 진행한다. "미해결 / 확인 필요"에 남긴 항목은 구현 후에도 이슈에 열어둔 채 보고한다.
