# toss-porter 구조 재설계 — 설계 문서

> 상태: 초안 (구현 순서 미정 — 사용자 선택 대기)
> 목적: toss-porter 할루시네이션 원인 중 "단일 대형 서브에이전트 일괄 처리" 구조를 어떻게 바꿀지 결정하기 위한 비교 문서.

---

## 1. 현재 구조 (확인된 사실)

- `toss-porter.md` = 단일 서브에이전트, 약 2,400줄 (tools: Read, Bash, Edit, Write, Agent).
- 16개 작업 단계 + 사전확인 + 검증을 한 번의 서브에이전트 실행으로 처리.
- 본문에 완성형 C# 패턴 ~2,000줄 상주 (서버시간·로그인·광고 pre-load·IAP·저장/불러오기·SafeArea·치트 등).
- AskUserQuestion / `❓판단 필요` / `👤사람 결정` 지점이 16단계 전반에 분산.
- 단계 의존: `3 로그인 → 3-A 로그`, `5 광고 → 6 IAP → 7 저장`은 순차 필수. 병렬 가능 그룹은 1·2·4 정도로 제한적.

### 이미 적용된 선행 수정 (1·2번)

- **1번 (완료):** 본문 코드 예시의 프로젝트 고유 식별자 5종(`EventEx.On<RESULT_CODE>`, `RESULT_CODE.SUCCESS`, `TimeManager.LocalNow`, `SaveLocal/LoadLocal`, `TreepllaNative.Vibrate`)을 `/* 예: … */` 주석으로 강등.
- **2번 (완료):** 진입점 1-V단계에 VOCAB 완전성 게이트 추가 — 미완 시 grep fallback 대신 중단·재스캔 요구. 실제 VOCAB 3종으로 검증.

---

## 2. 재설계가 풀어야 할 문제

| # | 문제 | 근거 |
|---|---|---|
| P1 | 템플릿 코드 leak | 1번에서 5종 확인 (구조적 위험 잔존) |
| P2 | 단일 컨텍스트 희석 | 16단계×2000줄 동시 상주, 앞 단계 Read 코드가 뒤로 밀림 |
| P3 | 중간 검수 불가 | 서브에이전트는 최종 메시지만 반환 → 사람이 단계별로 못 끼어듦 |
| P4 | 상호작용 부적합 | AskUserQuestion 다수가 격리 실행 안에서 흐름 끊김 |

---

## 3. 서브에이전트 vs 스킬 — 개념·장단점

둘 다 "토스 연동 해줘"로 시작될 수 있어 혼동되지만, 본질 차이는 **어디서 실행되느냐**다.

| 구분 | 서브에이전트 (현재) | 스킬 / 슬래시 명령 |
|---|---|---|
| 정의 위치 | `.claude/agents/porter/toss-porter.md` | `.claude/commands/h5/*.md` |
| 트리거 | 메인 루프가 Agent 도구로 위임 | 사용자가 `/toss-port` 타이핑, 또는 메인 루프가 Skill 도구로 호출 |
| 실행 컨텍스트 | 격리된 별도 컨텍스트 창 | 메인 대화 안에서 직접 실행 |
| 사용자 가시성 | 최종 메시지만 메인으로 반환 — 중간 단계 안 보임 | 모든 단계·질문·도구 호출 실시간 노출 |
| 중간 개입 | 어려움 (끝나야 결과 확인) | 쉬움 (단계마다 끼어들고 멈추고 수정) |
| 메인 컨텍스트 오염 | 없음 (격리) | 있음 (길어짐) |
| 병렬 | 가능 | 스킬 자체로는 안 됨 |

**핵심:** "B = 내가 호출"이 본질이 아니다. 본질은 **"보이고 끼어들 수 있다"**. 사용자가 겪은 "사람이 수정할 부분이 부족" = 격리 실행이라 중간이 안 보이기 때문(P3·P4). 스킬은 메인 대화에서 한 단계씩 진행되므로 그 자리에서 검수·수정이 된다.

## 4. 세 가지 안

### 안 C — 코드 패턴 분리 (영향도 낮음)

본문 코드 블록을 단계별 참조 파일로 분리.

```
porter/
  toss-porter.md            # 절차·판정·grep·AskUserQuestion만 (~600줄로 축소)
  patterns/
    toss-login.md           # 3 로그인
    toss-ad.md              # 5 광고 (on-demand/pre-load)
    toss-iap.md             # 6 IAP
    toss-save.md            # 7 저장/불러오기/치트
    toss-misc.md            # 햅틱·SafeArea·랭킹·공유·프로모션
```

각 단계에서 해당 패턴 파일을 그 시점에만 Read.

- 해결: P1(상주 코드 ↓), P2(상주 컨텍스트 ↓). 미해결: P3, P4.
- 마이그레이션: ① patterns/ 생성 ② 본문 코드 블록 이동 ③ 본문 각 단계를 "VOCAB 확인 → patterns/xxx.md Read → 적용" 한 줄로 치환.
- 위험: 낮음. 흐름 동일, 파일만 분리. 롤백 쉬움. **서브에이전트/스킬 어느 쪽이든 적용 가능.**

### 안 B — 대화형 스킬 전환 (중간)

서브에이전트 → 슬래시 명령(스킬)로 전환, 메인 대화에서 단계별 실행.

```
commands/h5/toss-port.md   # 메인 루프에서 단계별 진행
```

단계마다 AskUserQuestion·중간 결과 출력 → 사람이 검수 후 다음 단계.

- 해결: P3, P4. 부분 해결: P2(순차로 길어질 수 있음). 미해결: P1(C 병행 필요).
- 마이그레이션: ① 본문을 commands/로 이전 ② "일괄 실행" 전제를 "단계별 대화 진행"으로 문구 수정 ③ h5-port 오케스트레이터에서 호출 방식 변경.
- 위험: 중간. h5-port 연동·호출 습관 변경.

### 안 A — 오케스트레이터 + 단계별 서브에이전트 (높음)

toss-porter = 얇은 코디네이터. 단계(또는 파일 비겹침 그룹)별 소형 서브에이전트 호출. 각 서브에이전트는 해당 VOCAB 행 + 해당 패턴만 받음.

- 해결: P1·P2(격리 소형 컨텍스트), 파일 비겹침 병렬. 주의: P3는 코디네이터가 단계 사이 사람 체크포인트를 명시해야 확보. P4는 코디네이터가 질문 취합 필요.
- 마이그레이션: ① VOCAB 행 → 단계 라우팅 규칙 ② 단계별 서브에이전트 정의 ③ 코디네이터 작성 ④ 병렬 그룹/worktree 정합.
- 위험: 높음. 인프라·라우팅 복잡, 디버깅 어려움. 단계 의존(5→6→7) 많아 병렬 이득 제한적.

## 6. 권장 경로

**C → B 순차 적용, A는 보류.**

- 사용자가 가장 크게 체감한 문제는 P3·P4(중간 검수·상호작용) → 이를 직접 푸는 것은 **B(스킬)**. 단 B만으로는 P1·P2가 남으므로 **C를 먼저 깔고 B를 올림**.
- A의 주 이점인 병렬은 파이프라인 순차 의존(3→3-A, 5→6→7) 때문에 이득이 작고, 위험·복잡도는 가장 큼 → 후순위.

> C는 서브에이전트/스킬 어느 쪽이든 적용 가능하므로, C를 먼저 하면 B 전환 시 코드 패턴이 이미 외부 파일로 빠져 있어 전환이 가볍다.
