설치 & 업데이트

설치 (각자 1회)

cd <원하는 경로>/h5-porting-workflow
./install.sh
# 이후 Claude Code 재시작

업데이트 (git으로 관리한다면)

cd <원하는 경로>/h5-porting-workflow && git pull

이미 심볼릭된 기존 파일은 git pull만으로 즉시 최신화. templates/도 repo 실파일이라 pull하면 바로 최신.

주의: 이미 포팅한 프로젝트에 복사된 Editor/Runtime 스크립트는 자동 갱신 안 됨(복사본) — 갱신하려면 porting-init 재실행 또는 수동 재복사.

원리 — 심볼릭 등록 install.sh는 claude/ 폴더의 파일을 하나씩 ~/.claude/에 심볼릭 링크한다. Claude Code는 ~/.claude/agents·commands·skills를 읽으므로, 이 심볼릭 덕분에 agent·skill이 로컬 Claude Code에 등록되어 실행 가능해진다.
컴파일 체크 hook 추가 등록 필요: install.sh가 compile-check.sh를 ~/.claude/hooks/에 심볼릭까지는 해주지만, ~/.claude/settings.json의 hooks.PostToolUse에 직접 등록해야 .cs 저장 시 자동 컴파일 체크가 동작한다(jq 필요). 미등록 시 Tools/H5/Compile Check 메뉴를 수동 실행 — h5-port STEP 1-B가 등록 여부를 확인해 안내한다.
⚠ install.sh 재실행이 필요한 경우
  • 워크플로우를 처음 설치할 때
  • 새 agent·command·skill 파일이 추가됐을 때 (git pull 이후)

이유: install.sh는 실행 시점에 있는 파일만 심볼릭으로 연결한다. 나중에 추가된 파일은 다시 실행해야 인식된다.

메인 파이프라인 — /h5:h5-port 하나로 오케스트레이션

시작 전 필수: install.sh로 최초 1회 초기화가 되어 있어야 아래 파이프라인을 실행할 수 있다 (설치 방법은 설치 & 업데이트 슬라이드 참고).

체크리스트 확인 시점 — 타임라인

1. 스캔 완료한 후

{platform}-checklist.md 최초 생성, ## 확인 필요 항목 표시

↓

2. 포팅 시작 전 사람 확인

  • 물어볼지 말지 판단 기준 — PORTING_VOCAB.md의 ## Toss 전용(배너·프로모션), {IAP_METHOD} 행(IAP) 존재 여부
  • 확인 항목 — 배너 위치·IAP PID 매핑·프로모션 ID
  • 기록 위치 — toss-checklist.md / platform-checklist.md → ## 확인 필요
↓

3. 퓨어웹/플랫폼 포팅 진행 중

포터가 처리하며 ## 단계 진행 갱신(- [x]), 발견한 이슈는 ## 이슈에 기록

↓

4. 포팅 검증 완료한 후

포터의 완료 보고만 보고 넘어가지 않고, 체크리스트에 검증 결과가 실제로 기록됐는지 직접 재확인

↓

5. 전체 포팅 완료한 후 사람 확인

⏭️ 스킵 상태로 남은 판단 항목이 있으면 /platform-decisions 실행을 안내

햅틱 랭킹 버튼 공유 문구 UID/version 불필요 UI 삭제 로컬라이제이션

h5-port 후 사람이 결정해야 할 것: /platform-decisions

subagent(포터)는 실행 중 사용자에게 질문할 수 없다.
→ 사람 판단이 필요한 6개 항목은 체크리스트에 기록 후 platform-decisions skill로 이관.

③ 실제 플레이하며 햅틱 추가·메모 갱신
④ 만족스러운 결과가 나올 때까지 ③ 반복

워크플로우 자세히 설명

아래부터는 /h5:h5-port 명령어 하나로 자동 진행되는 STEP 0~4의 내부 동작을 단계별로 설명한다.
실행에는 이 명령어 하나면 충분하며, 이후 슬라이드는 세부 동작이 궁금할 때만 참고한다.

STEP 0 — 프로젝트 초기 설정

0-AHyperLane SDK 설치 확인

미설치면 AskUserQuestion으로 설치 여부를 확인한 뒤 hyperlane-init.sh 실행. 토스는 npx hyperlane setup toss로 wrapper 셋업 추가 필요.

0-Bporting-init 실행

CLAUDE.md· FRAMEWORK_REFERENCE.md 등 프로젝트 문서 생성.

0-C브랜치 확인

브랜치명이 {플랫폼}/{버전} 형식인지 확인. 형식이 아니면 사람 확인 대기 — 전환 후 계속 진행.

0-D산출물 커밋

CLAUDE.md·Docs/·Editor 툴 4종(CompileChecker 등) 커밋.

완료 신호: 커밋 성공 → STEP 1로 자동 진행

STEP 1 — 인코딩 변환 & 컴파일 확인

EUC-KR → UTF-8 변환

대상 파일 탐지 후 일괄 변환. 변환된 파일의 한글이 정상인지 사람이 샘플 확인.

1-AAndroid 컴파일 체크

오류 발생 시 파일 누락/SDK 문제/기타로 분류해 대표 3건만 보고. 분류별로 사람에게 처리 방안 확인.

1-BPostToolUse hook 확인

.cs 수정 시 자동 컴파일 체크 hook 존재 여부 확인. 없으면 수동으로 Tools/H5/Compile Check 메뉴 실행.

변환·수정한 파일은 각 단계 완료 시 즉시 커밋

STEP 2 — 프로젝트 분석

2-Aporting-scan

NATIVE_BASELINE·PORTING_VOCAB·기반 체크리스트(pureweb/toss) 생성.

2-Bporting-scan-verify

2-A 산출물 검증.

2-B-commit

분석 산출물 커밋. 광고 중 게임 일시정지 필요 여부 사람 확인.

2-D포팅 전 준비 체크리스트 보고

Trello 카드 등록·정상 작동 테스트 등 사람이 직접 처리해야 할 항목 포함.

2-E포터 실행 전 WebGL 컴파일 체크

오류 있으면 이슈로 기록 후 포터가 처리하도록 위임.

IAP/IAA/저장 역기획 문서(iap-analyzer 등)는 메인 시퀀스와 별개 — 필요할 때 개별 호출

STEP 3 — 포터 선택 및 실행

3-A사람 준비 항목 확인

baseline 측정 여부, 배너 위치·IAP PID 매핑·프로모션 ID 등을 AskUserQuestion으로 한 번에 확인(미확정도 허용).

3-B포팅 이슈 생성

GitHub remote 있으면 스텝별 이슈 생성(중복 방지).

3-C포터 실행

pureweb-porter → platform-porter → (토스면) 플랫폼별 porter 순서로 Agent 도구 실행. 완료 후 바로 STEP 4로.

포터는 subagent라 실행 중 질문 불가 — 판단 필요 항목은 체크리스트에 기록만 하고 계속 진행

STEP 4 — 포팅 검증

porting-verify 스킬 호출

퓨어웹 / 플랫폼공통(HLSDK) / 토스 각각 최종 검증.

검증 기록 교차 확인

체크리스트 ## 단계 진행의 검증 행이 실제 결과로 채워졌는지 확인(포터 자기보고 미신뢰 원칙).

4-A포팅 이슈 종료 처리

완료·확인 필요 없음 스텝만 개별 close, 미완료는 open 유지.

완료 보고에 남은 판단 항목이 있으면 /platform-decisions 실행 안내로 마무리

독립 실행 — 파이프라인 밖에서 직접 호출

skill / subagent용도
h5:stats-logging-analyzer (skill)여러 프로젝트의 agent-stats.md 대조 → 같은 (subagent, 라벨) 3회+ 반복 Zero-Hit 시 탐색 패턴 재검토 후보 보고 → 이슈화 제안
common:feature-breakdown → create-issue → resolve-issue (skill 체인)이 워크플로우 자체를 수정할 때 쓰는 계획 → 이슈 → 구현 루틴
기획자 전달용 역기획 문서를 만드는 content-scan/content-analyze·analyzer 3종(iap/iaa/save-point)은 별도 슬라이드 참고.
